# 抖音视频原文案提取自动化任务流程

版本：v0.1  
目标：将“抖音视频链接采集 → 豆包提取视频原文案 → 写回 Excel”的流程，从“RPA/剪贴板搬运”优化为“脚本与 Python 程序通过本地接口直接通信”。

---

## 1. 项目目标



```text
抖音视频链接
→ 进入 Python 程序任务队列
→ 豆包自动提取视频原生完整口播原文案
→ Python 自动写回 Excel 的“正文”列
```

最终产物是一份 Excel 表格，至少包含：

```text
序号｜标题｜链接｜正文
```

其中“正文”列由豆包提取结果自动补齐。

---

## 2. 原始流程

原流程如下：

```text
1. 打开抖音页面
2. 脚本 A 爬取一批视频链接
3. 脚本 A 导出 CSV，表头为：序号、标题、链接
4. 人工将 CSV 另存为 XLSX
5. 将 XLSX 导入 Python 程序
6. Python 程序读取表格，确保存在“正文”列
7. Python 程序复制当前“正文为空”的视频链接
8. RPA 将链接粘贴进脚本 B
9. 脚本 B 自动问豆包，并复制豆包回答
10. RPA 将豆包回答粘贴回 Python 程序输入框
11. RPA 点击「确认并下一个」
12. Python 将回答写入对应行“正文”列，并跳转到下一条
13. 重复直到全部处理完成
```

原流程能跑，但核心问题是：RPA 承担了大量“跨工具搬运数据”的工作。

---

## 3. 原流程的主要脆弱点

| 环节 | 问题 |
|---|---|
| CSV → XLSX | 需要人工另存，增加一步手工操作 |
| Python → 脚本 B | 依赖复制链接、粘贴链接 |
| 脚本 B → Python | 依赖复制回答、粘贴回答 |
| RPA | 需要判断焦点、窗口、按钮、剪贴板内容 |
| 剪贴板 | 容易被覆盖、复制失败、复制到旧内容 |
| 浏览器页面 | 可能弹窗、结构变动、按钮失效 |
| Excel 文件 | 如果被 WPS/Excel 占用，Python 保存可能失败 |

因此优化方向不是继续给剪贴板/RPA 打补丁，而是让工具之间通过接口直接传输结构化数据。

---

## 4. 优化后的整体架构

优化后角色划分如下：

```text
脚本 A：抖音采集器
Python 程序：数据中心 / Excel 管理器 / 任务队列 / 本地接口服务
脚本 B：豆包执行器
RPA：启动器 / 看门狗 / 异常兜底
```

优化后的主流程：

```text
脚本 A 采集视频链接
→ 批量 POST 给 Python 程序
→ Python 程序写入 Excel，并生成待处理任务
→ 脚本 B 从 Python 程序领取任务
→ 脚本 B 发给豆包提取正文
→ 脚本 B 把正文 POST 回 Python 程序
→ Python 程序写入对应行“正文”列
→ 脚本 B 继续领取下一条任务
→ 直到没有待处理任务
```

---

## 5. 优化后的数据流

### 5.1 脚本 A → Python 程序

脚本 A 在抖音页面采集视频链接。

当满足以下任一条件时，向 Python 程序提交一次数据：

```text
1. 已采集满 10 条
2. 当前批次不足 10 条，但采集任务已经结束
```

提交接口：

```text
POST http://127.0.0.1:8765/videos
```

提交数据示例：

```json
{
  "source": "douyin_favorite",
  "batch_id": "20260610_001",
  "items": [
    {
      "序号": 1,
      "标题": "视频标题 A",
      "链接": "https://www.douyin.com/video/xxx"
    },
    {
      "序号": 2,
      "标题": "视频标题 B",
      "链接": "https://www.douyin.com/video/yyy"
    }
  ]
}
```

Python 程序收到后执行：

```text
1. 检查 Excel 是否存在
2. 检查表头是否包含“正文”
3. 如果没有“正文”列，则自动追加
4. 按“链接”去重
5. 新链接追加到 Excel
6. 正文为空的行标记为 pending
7. 刷新程序界面
8. 返回新增数量、重复数量、当前待处理数量
```

返回示例：

```json
{
  "ok": true,
  "received": 10,
  "added": 8,
  "duplicate": 2,
  "pending": 8
}
```

---

### 5.2 Python 程序 → 脚本 B

脚本 B 不再等待 RPA 粘贴视频链接，而是主动向 Python 程序领取任务。

领取接口：

```text
GET http://127.0.0.1:8765/task/next
```

Python 返回当前下一条“正文为空”的视频任务：

```json
{
  "ok": true,
  "task": {
    "row_id": 12,
    "序号": 8,
    "标题": "某个视频标题",
    "链接": "https://www.douyin.com/video/xxx",
    "prompt": "仅负责提取视频原生完整口播原文案，禁止进行总结、概括、解读等操作，如果无法提取说明即可。"
  }
}
```

脚本 B 将以下内容发送给豆包：

```text
https://www.douyin.com/video/xxx

仅负责提取视频原生完整口播原文案，禁止进行总结、概括、解读等操作，如果无法提取说明即可。
```

如果没有待处理任务，Python 返回：

```json
{
  "ok": false,
  "message": "没有待处理任务"
}
```

脚本 B 收到后停止运行。

---

### 5.3 脚本 B → Python 程序

豆包回答完成后，脚本 B 读取最新回答正文，并回传给 Python 程序。

提交接口：

```text
POST http://127.0.0.1:8765/task/result
```

提交数据示例：

```json
{
  "row_id": 12,
  "链接": "https://www.douyin.com/video/xxx",
  "status": "done",
  "正文": "这里是豆包提取出的视频原生完整口播原文案"
}
```

Python 程序收到后执行：

```text
1. 根据 row_id 定位 Excel 行
2. 校验该行“链接”是否等于回传链接
3. 如果一致，写入“正文”列
4. 如果不一致，则按“链接”重新查找对应行
5. 写入成功后，状态改为 done
6. 保存 Excel
7. 自动定位下一条正文为空的任务
8. 刷新程序界面
```

返回示例：

```json
{
  "ok": true,
  "message": "已写入正文列",
  "next_pending": 17
}
```

## 6. Python 程序需要提供的接口

### 6.1 健康检查

```text
GET /health
```

作用：检查 Python 程序是否运行。

返回示例：

```json
{
  "ok": true,
  "service": "douyin_text_extract",
  "version": "0.1.0",
  "file_loaded": true,
  "pending": 12,
  "running": 1,
  "done": 30,
  "failed": 2
}
```

### 6.2 批量提交视频

```text
POST /videos
```

来源：脚本 A。  
作用：批量提交抖音视频链接及标题信息。

### 6.3 领取下一条任务

```text
GET /task/next
```

来源：脚本 B。  
作用：领取下一条“正文为空 / pending”的任务。

### 6.4 提交处理结果

```text
POST /task/result
```

来源：脚本 B。  
作用：提交豆包提取结果，写回 Excel。

### 6.5 提交失败原因

```text
POST /task/fail
```

来源：脚本 B。  
作用：当豆包无法提取、页面异常、回答超时等情况发生时，将失败原因回传给 Python。

提交示例：

```json
{
  "row_id": 12,
  "链接": "https://www.douyin.com/video/xxx",
  "status": "failed",
  "error": "等待豆包回答超时"
}
```

### 6.6 查看状态

```text
GET /status
```

作用：给脚本/RPA/人工查看当前整体进度。

## 7. Excel 表格设计

### 7.1 输入表头

脚本 A 原始导出表头：

```text
序号｜标题｜链接
```

### 7.2 Python 程序自动补充列

如果不存在以下列，则自动追加：

```text
正文
处理状态
失败原因
重试次数
创建时间
更新时间
完成时间
```

### 7.3 推荐正式表头

```text
序号｜标题｜链接｜正文｜处理状态｜失败原因｜重试次数｜创建时间｜更新时间｜完成时间
```

其中，最终交付给人看的干净版可以只保留：

```text
序号｜标题｜链接｜正文
```

## 8. 状态设计

每条视频任务建议有明确状态。

| 状态 | 含义 |
|---|---|
| pending | 待处理，正文为空 |
| running | 已被脚本 B 领取，正在豆包处理中 |
| done | 已写入正文 |
| failed | 处理失败，可重试 |
| manual | 需要人工处理 |
| duplicate | 重复链接，不新增 |

状态流转：

```text
新链接进入
→ pending
→ 脚本 B 领取
→ running
→ 豆包提取成功
→ done
```

失败流转：

```text
running
→ failed
→ 重试
→ pending / running
→ done / manual
```

## 9. 关键可靠性规则

### 9.1 链接去重

Python 程序必须按“链接”去重。

规则：

```text
链接已存在：不新增
链接不存在：追加新行
```

可选规则：

```text
正文为空时，允许更新标题
正文已有时，不覆盖标题和正文
```

### 9.2 row_id + 链接双重校验

脚本 B 回传结果时必须同时传：

```text
row_id
链接
```

Python 写入前校验：

```text
如果 row_id 行的链接 == 回传链接：
    写入该行
否则：
    按链接重新查找
    找到则写入
    找不到则标记异常
```

这样可以避免虚拟列表、排序变化、并发任务导致写错行。

### 9.3 running 超时回收

脚本 B 领取任务后，如果崩溃或页面卡死，任务会停在 running。

因此 Python 程序需要规则：

```text
running 超过 N 分钟未回传结果
→ 改回 pending
→ 重试次数 +1
```

建议默认：

```text
10 分钟超时回收
最多重试 3 次
超过 3 次改为 manual
```

### 9.4 Excel 文件占用处理

如果 Excel/WPS 正打开目标文件，Python 保存可能失败。

建议规则：

```text
处理过程中不要用 Excel/WPS 打开目标文件
```

更稳的方案：

```text
导入原始文件
程序保存工作副本
任务完成后导出最终版
```

### 9.5 脚本 B 连续失败保护

脚本 B 不应无限死循环。

建议规则：

```text
连续失败 3 次：停止
单次运行最多处理 N 条：默认 10，可配置
没有待处理任务：停止
豆包页面结构异常：停止并提示
```

## 10. RPA 的新职责

优化后，RPA 不再负责搬运数据。

不再需要 RPA 做：

```text
复制视频链接
粘贴到脚本 B
复制豆包回答
粘贴回 Python
点击确认并下一个
```

RPA 新职责变成：

```text
1. 打开抖音页面
2. 启动脚本 A
3. 打开豆包页面
4. 启动脚本 B
5. 检查左下角状态标记
6. 异常时刷新页面
7. 必要时重启脚本或提醒人工
```

也就是说，RPA 从“数据搬运工”降级为“启动器 / 看门狗”。

---

## 11. 脚本 A 职责

脚本 A 只负责抖音页面采集。

职责：

```text
1. 扫描当前收藏夹 / 列表页视频
2. 提取：序号、标题、链接
3. 本地去重
4. 每 10 条提交一次给 Python
5. 采集结束时提交剩余不足 10 条
6. 显示提交结果：新增几条、重复几条、失败几条
```

脚本 A 不负责：

```text
写 Excel
调用豆包
处理正文
```

---

## 12. 脚本 B 职责

脚本 B 只负责豆包页面执行。

职责：

```text
1. 向 Python 请求下一条任务
2. 将视频链接 + 提示词发送给豆包
3. 等待豆包回答完成
4. 读取最新回答正文
5. 将回答回传给 Python
6. 继续领取下一条任务
7. 无任务时停止
```

脚本 B 不负责：

```text
管理 Excel
保存 CSV
决定哪些行需要处理
维护总任务表
```

---

## 13. Python 程序职责

Python 程序是数据中心。

职责：

```text
1. 导入 CSV / XLSX
2. 自动补齐“正文”等必要列
3. 管理任务状态
4. 提供本地接口
5. 接收脚本 A 提交的视频链接
6. 给脚本 B 分发待处理任务
7. 接收脚本 B 回传的正文
8. 写入 Excel
9. 保存文件
10. 显示日志和任务进度
11. 处理失败、重试、超时回收
12. 导出最终 Excel
```

## 14. 推荐开发顺序

不要一次性全部重构。建议分阶段。

### 阶段 1：Python 程序支持接口

目标：

```text
在现有 Py 程序中增加：
GET /health
GET /task/next
POST /task/result
```

先让脚本 B 能直接和 Python 通信。

收益：

```text
去掉“脚本 B → RPA → Python”的回答搬运
```

### 阶段 2：脚本 B 接入接口

目标：

```text
脚本 B 从 Python 获取视频链接
脚本 B 提取完成后把正文回传 Python
```

完成后流程变成：

```text
Python → 脚本 B → 豆包 → 脚本 B → Python
```

RPA 不再参与正文传递。

### 阶段 3：Python 支持直接导入 CSV

目标：

```text
脚本 A 导出的 CSV 不再需要人工另存为 XLSX
Python 直接读取 CSV 并生成工作表
```

收益：

```text
减少人工 CSV 转 XLSX
```

### 阶段 4：脚本 A 直接提交给 Python

目标：

```text
脚本 A 采集到链接后直接 POST /videos
```

完成后，CSV 可以保留为备份，不再是主流程必需品。

### 阶段 5：RPA 降级为看门狗

目标：

```text
RPA 只负责打开页面、启动脚本、异常刷新
```

## 15. 最终流程

最终目标流程：

```text
1. 打开 Python 程序，选择或创建目标 Excel。
2. Python 启动本地接口服务。
3. 打开抖音页面。
4. 脚本 A 自动采集视频链接。
5. 脚本 A 每满 10 条或采集结束时提交给 Python。
6. Python 将新链接追加到 Excel，正文为空的标记为 pending。
7. 打开豆包页面。
8. 脚本 B 自动从 Python 领取 pending 任务。
9. 脚本 B 将链接和提示词发送给豆包。
10. 豆包生成回答。
11. 脚本 B 读取回答正文，并回传给 Python。
12. Python 写入对应行“正文”列，状态改为 done。
13. 脚本 B 继续领取下一条。
14. 没有任务后停止。
15. Python 导出最终 Excel。
```

---

## 16. 最小闭环

如果先做最小闭环，只需要打通：

```text
Python GET /task/next
脚本 B 领取任务
脚本 B 发送豆包
脚本 B POST /task/result
Python 写回正文
```

这一步完成后，最脆弱的“豆包回答通过 RPA 粘贴回 Python”就可以取消。

最小闭环完成后，再处理脚本 A 直连 Python。
